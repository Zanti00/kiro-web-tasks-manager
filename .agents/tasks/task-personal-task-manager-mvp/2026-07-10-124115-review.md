# Personal task manager with Supabase auth and CRUD (v2)

Three of five v1 issues are resolved: signup now redirects to `/login?signup=success` instead of dead-ending, `TaskList` syncs with server-revalidated props via `useEffect`, and both auth forms enforce `minLength={6}` on the password input. The remaining two (redundant `user_id` in insert, middleware deprecation) were intentionally kept as-is per author decision.

**Watch for:** After `createTask` succeeds, the newly created task won't appear in the list until the server revalidation round-trip completes — `AddTaskForm` resets its input but doesn't optimistically inject the new task into `TaskList`'s state (likely). The `updateTask`, `deleteTask`, and `toggleTask` server actions don't verify row-affected count — RLS silently filters non-owned rows, so optimistic UI changes for those rows would never actually persist (confirmed).

**Verdict**: APPROVED

## High-level view

The signup flow now redirects to `/login?signup=success`, and the login page shows a success banner via `useSearchParams` wrapped in `Suspense`.

The stale-list fix (`useEffect` syncing `initialTasks`) works for page-level revalidation, but `createTask` has no optimistic path into `TaskList`. The form clears on success; the new task appears only once `revalidatePath` triggers a server re-render and React reconciles the new props through the effect. On slow connections this produces a visible gap between form reset and task appearance, potentially exceeding the PRD's "< 1s visible in list" target.

The CRUD server actions delegate authorization entirely to RLS, which means the application layer never distinguishes "task not found" from "access denied." Supabase returns `{ error: null, count: 0 }` when RLS filters a row, so the action reports success despite affecting nothing — causing optimistic UI state to diverge from persistence until the next revalidation.

## Create-task feedback gap

```tsx
startTransition(async () => {
  const result = await createTask(formData);
  if (result?.error) {
    setError(result.error);
  } else {
    formRef.current?.reset();
  }
});
```

On success the form clears, but the new task isn't added to `TaskList`'s local state — there's no callback, no shared context, no optimistic entry. The task appears when `revalidatePath("/")` causes the server component to re-render and the `useEffect` picks up new props. On slow connections or under load, the user sees a cleared form and an unchanged list for a noticeable interval, contradicting the PRD's "< 1s visible in list" requirement.

The fix is either an optimistic insert (requiring the component to predict the task's shape, including a temporary ID) or lifting the task list into a shared state that `AddTaskForm` can push into directly.

## Silent RLS-rejected mutations

```typescript
const { error } = await supabase.from("tasks").delete().eq("id", id);
if (error) { return { error: error.message }; }
```

When RLS filters out the row (wrong user, or ID doesn't exist), Supabase returns `{ error: null, count: 0 }`. The action returns `{ success: true }` despite affecting nothing. `TaskList` has already applied the optimistic update, so the UI shows the task as deleted/updated. On next revalidation (page navigation, another mutation), the original state reappears — a ghost update.

For a single-user MVP this is unlikely to surface in normal use, but a stale tab retaining task IDs that were deleted by another session would produce confusing state resurrection.
