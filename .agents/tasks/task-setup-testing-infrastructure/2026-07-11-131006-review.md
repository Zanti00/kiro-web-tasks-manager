# Vitest infrastructure and unit/component test suite for task manager

This change bootstraps a complete testing stack (Vitest 4.x, React Testing Library, jsdom) and ships 52 tests across six files covering all three server action modules and both interactive components. The test approach favors behavioral assertions — form submissions, optimistic UI, rollback on error — over implementation detail. The Supabase client is replaced by a hand-rolled mock object per file, and Next.js navigation/cache primitives are stubbed to capture redirects and revalidation calls.

Watch for: the Supabase mock in `actions.test.ts` uses a shared `mockEq` function across `update` and `delete` chains, which makes tests order-sensitive if parallelism changes (**likely**); `createTask` test doesn't assert that insert is *not* called when auth fails, missing a regression guard on operation ordering (**confirmed**); no tests exist for page-level components (`login/page.tsx`, `signup/page.tsx`, `page.tsx`) or the server-component data-fetching path (**confirmed**).

## High-level view

Server action tests mock the Supabase client at the module boundary (`@/lib/supabase/server`), intercepting the chained query builder methods. The shared `mockEq` across both `update` and `delete` chains means the test is structurally coupled to call order — a concern if test isolation ever weakens.

Component tests exercise the optimistic-update-then-rollback pattern that drives `TaskList` and the form-submission-with-error-display pattern in `AddTaskForm`. They verify user-observable behavior rather than internal state transitions, and test the rollback path on every mutation. A gap exists around rapid interaction (double-toggle before server responds).

Coverage gaps exist at the page level. The dashboard does auth-gated data fetching that silently renders an empty task list when the user is unauthenticated, and the login page's `useActionState` integration is untested — both paths where Next.js regressions commonly surface.

1. **Shared `mockEq` across update/delete chains** — The same `vi.fn()` instance handles `.eq()` for both `update` and `delete` in `actions.test.ts`. Split into `mockUpdateEq` and `mockDeleteEq` to eliminate implicit ordering dependency.

3. **No page-level component tests** — `login/page.tsx`, `signup/page.tsx`, and the dashboard server component have no test coverage. The login page's `useActionState` integration and the dashboard's auth-gated fetch are both behavioral paths worth testing.
4. **`createTask` test doesn't verify auth check ordering** — The test for "redirects when user is not authenticated" verifies the redirect happens, but doesn't assert that the Supabase insert was *not* called. A regression where insert fires before auth wouldn't be caught.
5. **Smoke test is low-signal** — `smoke.test.ts` verifies `1+1=2` and DOM matchers. It should be removed once real tests are passing, or it will add noise to every CI run indefinitely.

## Supabase mock topology and the shared `mockEq` problem

The mock in `actions.test.ts` constructs a fake Supabase client where `from()` returns an object with `insert`, `update`, and `delete` methods. The `update` and `delete` methods both return `{ eq: mockEq }`, meaning there is a single mock function tracking `.eq()` calls for both code paths. This creates a fragile coupling: if a developer adds a test that exercises both update and delete in sequence, the assertions on `mockEq` become ambiguous — you can't tell which code path triggered the call.

The fix is trivial — separate `mockUpdateEq` and `mockDeleteEq` — and would make assertions self-documenting: `expect(mockDeleteEq).toHaveBeenCalledWith("id", "task-1")` reads clearer than `expect(mockEq).toHaveBeenCalledWith("id", "task-1")` in a delete test.

## Auth assertion gap in createTask

The `createTask` tests cover the "redirect when unauthenticated" path by asserting the redirect mock throws. However, they don't assert that `mockInsert` was *not* called in that scenario. The production code checks auth before inserting, but if a future refactor reorders those calls (insert first, auth check second), the test would still pass. Adding `expect(mockInsert).not.toHaveBeenCalled()` to the unauthenticated test would close this gap.

## Rapid-toggle behavioral gap in TaskList

There is no test for what happens when a user rapidly toggles the same task (double-click before server responds). The optimistic state is applied synchronously via `setTasks`, so the second click's "previous value" is already the optimistic value, not the server-confirmed value. If the first server call then fails and triggers rollback, the component rolls back to the wrong state. A test documenting the expected behavior — whether it's "last write wins" or "first server response triggers reconciliation" — would prevent future confusion about the intended contract.

## Coverage boundary: page components

The pages contain behavioral logic beyond simple rendering:

- `login/page.tsx` uses `useActionState` to wire form submission and conditionally renders a signup success banner based on URL search params (`?signup=success`).
- `page.tsx` (dashboard) performs auth-gated data fetching — if `getUser()` returns null, it renders with `user?.email` as undefined and `tasks ?? []` as empty, silently showing an empty dashboard rather than redirecting.

These represent the integration layer where server actions meet rendering, and they're where regressions most often surface in Next.js apps.
