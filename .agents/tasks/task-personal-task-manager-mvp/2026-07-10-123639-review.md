# Supabase-backed personal task manager with email auth and RLS

A complete MVP for a personal task app: email/password auth via Supabase, a tasks table with row-level security, middleware that bounces unauthenticated requests to `/login`, and a server-rendered dashboard backed by client components with optimistic CRUD. The architecture splits cleanly — server actions for mutations, a server component for the initial fetch, and client components for interactive state.

**Watch for:** The `user_id` column is populated from a client-supplied `getUser()` call rather than relying exclusively on the RLS policy's `auth.uid()`, which means the server action's auth check and the insert are not atomic (likely). Signup succeeds but doesn't redirect or sign the user in, leaving them on a success message with no clear path to the dashboard (confirmed). The task list is initialized once at render time and never re-synced with the server after mutations from other tabs or after the `revalidatePath` that the server actions call (confirmed).

## High-level view

Auth flow relies on middleware to redirect unauthenticated users and refresh the session on every request. Login calls `signInWithPassword` and redirects home. Signup calls `signUp` and stops at a confirmation message — it doesn't establish a session, so a user who signs up with email confirmation disabled will have to manually navigate to login.

The dashboard server component fetches tasks once, then hands them to `TaskList` which manages its own local copy with optimistic updates. Server actions call `revalidatePath("/")` after mutations, but because `TaskList` initializes state from `initialTasks` only on mount, the revalidation's new data will only appear after a full page navigation — not after the action returns.

The CRUD actions validate input and handle errors, but `createTask` manually inserts `user_id` from `getUser()` instead of using a database default. This is redundant with RLS's `with check` and introduces a subtle race if the session expires between the `getUser()` call and the insert.

1. **Signup dead-end** — After successful signup, the user sees a static message with no redirect and no session. Either redirect to login, or auto-sign-in and redirect to dashboard. (confirmed)
2. **Stale task list after mutation** — `TaskList` initializes `useState` from props once and never picks up server-side revalidation. Use `useEffect` to sync when `initialTasks` changes, or use `key` prop reset, or switch to `useOptimistic` with the Next.js streaming pattern. (confirmed)
3. **Redundant `user_id` in insert** — `createTask` explicitly sets `user_id` from `getUser()`. If the session expired between the auth check and the insert, RLS would reject it anyway, but the non-atomic pair creates a window where the auth check passes and the insert uses a stale user reference. Rely on a Postgres default (`default auth.uid()`) or a trigger instead. (likely)
4. **No password length enforcement** — Neither login nor signup actions validate password length. Supabase enforces a minimum server-side, but the error is opaque. Add client-side `minLength` on the password input for a better UX. (confirmed)
5. **Middleware deprecation** — Next.js 16 deprecates the `middleware.ts` convention in favor of `proxy`. This works today but will break on a future Next.js upgrade. (confirmed)

## Signup flow dead-end

The `signup` server action calls `supabase.auth.signUp` and, on success, returns `{ success: "Check your email to confirm your account." }`. The UI renders this as a green banner, but there's no redirect and no automatic sign-in. If the Supabase project has email confirmation disabled (a common dev-mode setting), the user has a valid account and session cookie already issued by the `signUp` call but is left staring at the success message with no navigation to the dashboard.

The fix is either: (a) detect the `session` returned from `signUp` and redirect if present, or (b) after the success message, redirect to `/login` with a flash message.

## Task list state synchronization

```tsx
// TaskList.tsx
const [tasks, setTasks] = useState<Task[]>(initialTasks);
```

This captures the server-rendered task list at mount time. After a mutation, the server action calls `revalidatePath("/")`, which tells Next.js to re-fetch the page data. But since `TaskList` is a client component that already mounted, React will not reinitialize its state from the new props — `useState` only uses the initial value on first render.

The consequence: if the optimistic state diverges from reality (e.g., the server action fails silently, or another tab mutated data), the list won't self-correct until a full page reload.

Two standard fixes: pass a `key` prop that changes on mutation (forces remount), or add `useEffect(() => setTasks(initialTasks), [initialTasks])` to pick up server revalidations.

## Auth check and RLS redundancy in createTask

```typescript
// app/actions.ts
const { data: { user } } = await supabase.auth.getUser();
if (!user) { redirect("/login"); }
const { error } = await supabase.from("tasks").insert({ title, user_id: user.id });
```

The `user_id` column has no `default auth.uid()` in the migration, so the explicit value is necessary for the insert. But the two-step pattern (check auth, then insert with user data) isn't atomic. The cleaner pattern is adding `default auth.uid()` to the `user_id` column in the migration, then omitting `user_id` from the insert entirely. This eliminates the possibility of a bug where the wrong user ID is inserted and removes the need for the manual auth guard on insert.

## Missing password validation UX

The HTML inputs use `required` but not `minLength`. Supabase rejects passwords shorter than 6 characters with a generic error message. Adding `minLength={6}` to the password inputs would prevent the round-trip and provide immediate feedback.
